# Musculata Bot — MVP

## Как запустить в PyCharm

1. Открой эту папку как проект в PyCharm.
2. Создай виртуальное окружение (PyCharm предложит сам при открытии, либо
   `File → Settings → Project → Python Interpreter → Add Interpreter`).
3. Установи зависимости:
   ```
   pip install -r requirements.txt
   ```
4. Скопируй `.env.example` в `.env` (просто переименуй копию) и впиши туда свой
   токен бота:
   ```
   BOT_TOKEN=твой_настоящий_токен
   ```
5. **Перевыпусти токен через @BotFather**, если старый токен уже где-то
   светился (например, в чате) — старый нужно считать скомпрометированным.
6. Запусти `main.py`.

## Что уже работает

- Главное меню в стиле "Ордена" — 2 строки по 3 кнопки. Создаётся
  **ровно один раз за всю историю общения** с конкретным пользователем
  (при самом первом `/start`) и больше никогда не пересоздаётся и не
  удаляется — см. ниже, почему это принципиально для отсутствия "мигания".
- `/start` зарегистрирован в системном списке команд Telegram (иконка "/"
  у поля ввода).
- Каталог организован по категориям (данные были в самой таблице
  СМЕТА_ДЛЯ_БОТА.xlsx, столбец "Категория" — 19 дробных категорий
  объединены в 9 осмысленных групп).
- При добавлении товара в корзину кнопка на карточке товара сама меняется:
  "➕ Добавить в корзину" → "✅ В корзине: N шт." (Telegram не поддерживает
  цветные кнопки — галочка это ближайшая имитация "загорелась зелёным").
- Каждое сообщение, которое пользователь отправляет нажатием кнопки меню,
  бот сразу удаляет — в чате остаётся только актуальный "экран".
- При оформлении заказа показывается сумма без скидки, скидка 10% за
  использование бота и итоговая цена.

## Почему меню больше никогда не мигает и не съезжает

Раньше кнопка "Начать сначала" при каждом нажатии удаляла старое меню и
создавала новое — это и есть тот самый видимый "мигающий" эффект в
Telegram (клавиатура на долю секунды пропадает и появляется заново).
Теперь:

1. Кнопка "Начать сначала" убрана полностью.
2. В `main.py` меню создаётся **один-единственный раз** — при самом первом
   `/start` пользователя. Дальше, даже если человек отправит `/start`
   повторно, код проверяет `state.get_menu(user_id)`: если меню уже
   существует, оно не трогается вообще — вместо этого показывается просто
   приветственный текст поверх текущего "экрана".
3. Все остальные экраны (каталог, категории, корзина, заглушки) устроены
   так, что в принципе физически не могут задеть меню: они никогда не
   отправляются с `ReplyKeyboardMarkup`, а удаление/пересоздание
   происходит только в рамках отдельного "контентного" сообщения
   (`state.content_message`), полностью независимого от сообщения с меню
   (`state.menu_message`).

Итог: меню трогается кодом ровно один раз в жизни каждого пользователя.
Никакое нажатие любой другой кнопки на него физически повлиять не может.

## Безопасность и устойчивость — что сделано и что стоит знать

### Что уже реализовано в этой версии

1. **Секреты не в коде.** Токен бота и (опционально) ID администратора —
   только в `.env`, который в `.gitignore` и никогда не попадёт в Git.
2. **Rate limiting.** Не больше 15 действий за 10 секунд от одного
   `telegram_id` (настраивается в `config.py`) — защита от спама и флуда.
   При превышении бот просто молча игнорирует лишние нажатия, не отвечая
   новыми сообщениями (чтобы не подпитывать сам флуд).
3. **Обработка ошибок в каждом обработчике.** Декоратор `safe_handler`
   (в `errors.py`) оборачивает вообще все обработчики сообщений и
   кнопок — если внутри случится необработанное исключение, оно не уронит
   весь бот и не зависнет молча: ошибка логируется и сразу летит
   уведомлением тебе в Telegram.
4. **Уведомление в Telegram при падении.** Если задать `ADMIN_CHAT_ID` в
   `.env` (как получить — см. `.env.example`), при любой ошибке в
   обработчике или при полном падении бота ты получишь сообщение с
   текстом ошибки, а не узнаешь о проблеме только когда сам заметишь.
5. **Логи в файл с автоочисткой.** `logging_setup.py` пишет логи в
   `logs/bot.log` с ротацией: как только файл достигает 2 МБ, он
   архивируется (`bot.log.1`, `bot.log.2`...), после 5 файлов самый
   старый удаляется автоматически. Логи никогда не займут больше ~12 МБ
   на диске. Заодно приглушено избыточное ERROR-логирование самой
   библиотеки при обычной остановке бота (см. предыдущий разговор).
6. **Аналитика лидов в CSV.** `analytics.py` пишет в `analytics.csv`:
   кто написал `/start`, какие категории смотрел, какие товары открывал,
   что добавлял в корзину, кто дошёл до оформления. Открывается как
   обычная таблица (Excel/Google Sheets) или через `pandas.read_csv()`.
   Хранятся только `telegram_id` и публичный username — никаких имён,
   телефонов, email.

### Что стоит держать в голове дальше (не решается только кодом)

- **Если процесс бота убьёт сама операционная система** (например,
  сервер перезагрузился, закончилась память) — никакой код внутри
  Python-скрипта это не поймает и не пришлёт уведомление, потому что сам
  процесс уже не выполняется. Для настоящей отказоустойчивости на
  сервере стоит запускать бота через супервизор, который сам
  перезапускает процесс при падении: на Linux — `systemd`-сервис с
  `Restart=on-failure`, либо `supervisor`/`pm2`; на Windows — Планировщик
  заданий с перезапуском при сбое. Это отдельная настройка на уровне
  хостинга, не в коде бота.
- **Корзина и состояние экранов — в памяти процесса.** Перезапуск бота
  обнуляет и то, и другое. Для продакшена (важно не терять данные) —
  следующий шаг это база данных (SQLite/PostgreSQL), обсуждали раньше.
- **In-memory-хранилища не защищены блокировками (locks).** Если два
  запроса от одного пользователя придут строго одновременно (очень
  редкий случай), возможен минорный рассинхрон счётчика корзины. При
  переходе на базу данных эта проблема снимается сама собой транзакциями.
- **Регулярно обновляй зависимости** — `pip list --outdated` раз в
  квартал, особенно `pyTelegramBotAPI`, там могут закрывать уязвимости.
- **Никогда не публикуй `analytics.csv` и `logs/`** — они бытовые
  (не содержат явных персональных данных), но всё равно это данные о
  поведении реальных пользователей, есть смысл держать их не в публичном
  доступе.

## Про ошибки "Infinity polling: polling exited" / "Break infinity polling"

Это служебные сообщения самой библиотеки `pyTelegramBotAPI` — она пишет
их в лог на уровне ERROR **каждый раз**, когда цикл polling'а завершается,
включая обычную ручную остановку скрипта (Ctrl+C). Само по себе это не
значит, что бот сломался — просто выбор уровня логирования у библиотеки
довольно "нервный" по умолчанию.

В `main.py` уровень логирования библиотеки приглушён
(`logging.getLogger("TeleBot").setLevel(logging.CRITICAL)`), а сам запуск
обёрнут в `try/except KeyboardInterrupt`, так что при обычной остановке
через Ctrl+C ты увидишь просто "Бот остановлен вручную." без пугающих
ERROR-строк.

Если такие ошибки будут появляться **сами по себе, во время работы бота**
(а не при ручной остановке) — это признак реального обрыва соединения
(нестабильный интернет/прокси/VPN) или необработанного исключения внутри
одного из обработчиков. `infinity_polling()` в норме сам переподключается
после сетевого сбоя, так что единичные обрывы не страшны; если это
происходит часто — стоит посмотреть трейсбек прямо над этими двумя
строками, там будет настоящая причина.

## Согласие на обработку персональных данных — жёсткий вариант (с гейтом)

При первом `/start` бот отправляет официальное уведомление (текст в
`legal.py`, `CONSENT_TEXT`) с единственной кнопкой **"✅ Принимаю"**.
Пока пользователь её не нажмёт:

- меню внизу вообще не создаётся;
- любая попытка вызвать каталог/корзину/поддержку/другие разделы (даже
  если как-то обойти отсутствие кнопок и прислать текст руками) блокируется
  — бот просто присылает напоминание нажать "Принимаю", и ничего не выполняет.

Это реализовано не точечными проверками в каждом обработчике, а на уровне
самого декоратора `safe_handler` (`errors.py`): по умолчанию он требует
согласия для ЛЮБОГО обработчика, и это можно отключить явно —
`@safe_handler(bot, require_consent=False)` — только для трёх мест, где
это осознанно не нужно: `/start` (иначе согласие неоткуда взять), сама
кнопка "Принимаю" (иначе на неё же и не нажать) и обработчик Reply-ответов
администратора (это не покупатель, гейтить нечего).

После нажатия "Принимаю":
- кнопка исчезает из уведомления, текст помечается "✅ Согласие получено";
- само уведомление остаётся в чате навсегда — как юридический текст, не
  трогается ни при каких дальнейших действиях;
- сразу же создаётся обычное меню, дальше всё работает как раньше.

Проверено вживую (на симуляции): попытка открыть каталог до согласия
блокируется, после нажатия "Принимаю" — открывается без проблем.

## Поддержка — теперь полноценная двусторонняя связь с администратором

Раньше кнопка "Поддержка" была заглушкой. Теперь:

1. Пользователь жмёт "⚒️ Поддержка" → бот просит написать вопрос одним
   сообщением.
2. Следующее текстовое сообщение пользователя (если это не нажатие другой
   кнопки меню) пересылается администратору (`ADMIN_CHAT_ID`) вместе с
   именем, @username и `user_id` пользователя.
3. **Администратор отвечает обычным Reply** прямо на пересланное
   сообщение в Telegram — бот автоматически определяет, какому
   пользователю адресован ответ, и отправляет текст ему. Администратору
   не нужно ничего копировать вручную или знать команды.
4. Если пользователь передумал (нажал любую другую кнопку меню, не
   дописав вопрос) — ожидание вопроса автоматически отменяется.

Ограничение текущей версии: для каждого нового вопроса пользователю нужно
заново нажать "Поддержка" — бот не воспринимает случайные последующие
сообщения как продолжение того же диалога. Это осознанное упрощение,
чтобы не путать вопросы поддержки с обычной перепиской.

## Если уведомления администратору не приходят

При каждом запуске (`python main.py`) бот теперь сам проверяет, реально
ли долетает сообщение до `ADMIN_CHAT_ID`, и прямо пишет результат в
консоль:

- **`✅ Тестовое сообщение отправлено...`** — всё настроено верно.
- **`⚠️ НЕ удалось отправить...`** — с текстом причины. Две самые частые:
  1. **Аккаунт, чей ID указан в `ADMIN_CHAT_ID`, ни разу не писал `/start`
     этому конкретному боту.** Telegram запрещает ботам первыми писать
     пользователям, которые никогда не начинали с ними диалог — это
     защита от спама на уровне платформы, не баг бота. Решение: зайди в
     Telegram под тем аккаунтом, который должен получать уведомления, и
     нажми `/start` этому боту хотя бы один раз.
  2. **В `ADMIN_CHAT_ID` указано не то значение.** Там должен быть
     числовой ID (например, `123456789`), а не `@username` и не номер
     телефона. Получить свой числовой ID: напиши `@userinfobot` в
     Telegram, он пришлёт в ответ поле `Id:` — это и есть нужное число.

Если в `.env` `ADMIN_CHAT_ID` вообще не задан — при старте бот прямо об
этом напишет в консоли, чтобы не гадать, почему уведомления не приходят.

## Про сброс категорий/страниц в callback_data

Кнопки товаров и навигации кодируют состояние в самих себе (например,
`cat:3:1` — категория №3, страница 1). Если бот перезапускается, а
пользователь нажимает кнопку из старого (созданного до перезапуска)
сообщения — это не сломается: категория ищется по индексу в списке
`CATEGORIES`, который пересчитывается из `products.json` при каждом
старте в одном и том же порядке.

## Как это устроено технически (если будешь дорабатывать)

В `state.py` теперь два независимых "якоря" на пользователя:
- **menu_message** — id сообщения с постоянным меню внизу. Устанавливается
  один раз в `/start`, дальше код его никогда не удаляет и не пересоздаёт.
- **content_message** — id текущего "экрана" (каталог/корзина/заглушка).
  Именно он удаляется и пересоздаётся при каждом переходе, через функцию
  `show_content()` в `main.py`.

Это разделение и обеспечивает то, что меню всегда на месте: сообщения с
контентом не несут `ReplyKeyboardMarkup`, поэтому их удаление никак не
затрагивает уже показанную клавиатуру — в Telegram кастомная клавиатура
привязана к чату, а не к конкретному сообщению.

## Важный нюанс про удаление сообщений

Бот удаляет свои предыдущие сообщения через `bot.delete_message`. Это работает
только для сообщений младше **48 часов** и только тех, что отправил сам бот
(так устроен Telegram Bot API). Если пользователь не писал боту больше двух
суток — при следующем переходе между разделами удаление старого сообщения
может не сработать, бот просто тихо это проигнорирует (ошибка перехвачена
в коде) и покажет новый экран рядом.

## Что НЕ реализовано (это отдельные, более крупные этапы)

1. **Корзина хранится в памяти процесса.** Перезапуск бота = все корзины
   обнуляются. Для продакшена нужна база данных (SQLite/PostgreSQL).
2. **Формат ссылки на оформление заказа — предположительный.** Нужно
   подтверждение от технического специалиста сайта MashinaBody: какой именно
   формат параметров реально принимает checkout (артикулы или внутренние ID,
   разделители, имя параметра для купона). Без этого подтверждения корзина
   на сайте может не заполниться.
3. **Реферальная система** (⚔️ Пригласить соратника) — нужна база данных
   для хранения кто кого пригласил, начисление бонусов, защита от повторного
   учёта одного и того же приглашённого.
4. **Орден / подписка на год** (🛡️ Вступить в Орден) — нужен приём платежей
   (Telegram Payments API или эквайринг), хранение даты оплаты, планировщик
   уведомлений раз в 60 дней, хранение и изменение адреса доставки.
5. **История заказов** — нужна БД, куда пишутся завершённые заказы
   (сейчас бот не узнаёт, когда оплата на сайте реально прошла — для этого
   нужен webhook от WooCommerce обратно в бота).
6. **Поддержка** — сейчас просто просит написать вопрос, реальная пересылка
   администратору не подключена.

Прежде чем разворачивать реферальную систему и Орден — стоит получить от
сайта ответ по формату checkout-параметров (пункт 2 выше). Это самый
